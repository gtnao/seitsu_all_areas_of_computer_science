# バッファオーバーフロー — メモリ破壊攻撃の原理と防御の歴史

## 1. 背景と動機

### 1.1 なぜバッファオーバーフローは危険なのか

バッファオーバーフローは、コンピュータセキュリティにおいて最も古く、最も破壊的な脆弱性クラスの一つである。1988年のMorris Wormから2020年代の現在に至るまで、数十年にわたってオペレーティングシステム、Webサーバー、ネットワーク機器、組み込みシステムなど、あらゆるソフトウェアで繰り返し悪用されてきた。

バッファオーバーフローとは、プログラムが確保したメモリ領域（バッファ）の境界を超えてデータを書き込んでしまう脆弱性である。この「境界を超える」という一見単純なバグが、なぜ任意のコード実行という壊滅的な結果をもたらすのか。それを理解するためには、プログラムがメモリ上でどのように動作しているかを深く知る必要がある。

### 1.2 C言語とメモリ安全性の欠如

バッファオーバーフローの根本的な原因は、C言語（およびC++）が提供する低レベルなメモリ操作と、境界チェックの欠如にある。C言語は1972年にDennis RitchieがUNIXオペレーティングシステムの開発のために設計した言語であり、「プログラマを信頼する」という哲学のもと、配列の境界チェックやメモリアクセスの検証を言語仕様として強制しない。

この設計判断には合理的な理由があった。1970年代のハードウェアでは、すべてのメモリアクセスに境界チェックを挿入することは許容できないパフォーマンスオーバーヘッドを生んだ。しかし、この「信頼」は悪意のある入力に対しては致命的な弱点となる。攻撃者は、プログラムが想定していない長さの入力を送り込むことで、メモリの内容を意図的に書き換え、プログラムの制御フローを乗っ取ることができる。

### 1.3 歴史的な位置づけ

バッファオーバーフローの危険性は、1996年にAleph One（Elias Levy）が発表した論文「Smashing the Stack for Fun and Profit」によって広く知られるようになった。この論文は、スタックバッファオーバーフローの原理と実際のexploitの構築方法を詳細に解説し、セキュリティコミュニティに衝撃を与えた。

しかし、バッファオーバーフロー自体はそれよりはるか以前から存在していた。1988年のMorris Wormは、UNIXの`fingerd`プログラムのバッファオーバーフローを悪用して拡散した最初の大規模なインターネットワームである。この事件は、ネットワーク接続されたシステムにおけるソフトウェアの脆弱性がもたらすリスクを世界に示した最初の事例となった。

## 2. プロセスのメモリレイアウト

バッファオーバーフローの仕組みを理解するためには、まずプロセスがメモリ上でどのように構成されているかを知る必要がある。典型的なx86/x86-64アーキテクチャにおけるプロセスのメモリレイアウトは以下の通りである。

```
高位アドレス (0xFFFFFFFF)
+---------------------------+
|        カーネル領域         |  <-- ユーザープロセスはアクセス不可
+---------------------------+
|         スタック (Stack)    |  <-- 局所変数、戻りアドレス、関数引数
|             |              |      高位アドレスから低位に向かって伸びる
|             v              |
|                            |
|          (空き領域)         |
|                            |
|             ^              |
|             |              |
|         ヒープ (Heap)       |  <-- malloc()等で動的に確保される領域
+---------------------------+      低位アドレスから高位に向かって伸びる
|         BSS セグメント      |  <-- 未初期化グローバル/static変数
+---------------------------+
|       Data セグメント       |  <-- 初期化済みグローバル/static変数
+---------------------------+
|       Text セグメント       |  <-- 実行可能なマシンコード（読み取り専用）
+---------------------------+
低位アドレス (0x00000000)
```

### 2.1 Text セグメント

プログラムのコンパイル済みマシンコードが格納される領域である。通常は読み取り専用（Read-Only）としてマッピングされ、実行時に変更されることはない。複数のプロセスが同一のプログラムを実行する場合、Textセグメントを共有することでメモリを節約できる。

### 2.2 Data セグメント

初期化済みのグローバル変数および`static`変数が格納される。プログラムのコンパイル時に値が確定しており、実行ファイル内にその初期値が含まれている。

```c
int global_var = 42;           // Data segment
static char msg[] = "Hello";   // Data segment
```

### 2.3 BSS セグメント

BSS（Block Started by Symbol）は、未初期化のグローバル変数および`static`変数が格納される領域である。実行時にゼロで初期化される。実行ファイルのサイズを削減するため、BSSセグメントにはデータの実体は含まれず、サイズ情報のみが記録される。

```c
int uninitialized_global;       // BSS segment (zero-initialized at runtime)
static int uninitialized_static; // BSS segment
```

### 2.4 ヒープ（Heap）

`malloc()`、`calloc()`、`realloc()`（Cの場合）や`new`（C++の場合）によって動的に確保されるメモリ領域である。低位アドレスから高位アドレスに向かって成長する。プログラマが明示的に解放（`free()`/`delete`）しない限りメモリは占有され続けるため、メモリリークの原因となる。

### 2.5 スタック（Stack）

関数呼び出しに伴うデータを格納する領域であり、バッファオーバーフロー攻撃において最も重要な標的となる。スタックは高位アドレスから低位アドレスに向かって成長する（x86/x86-64アーキテクチャの場合）。

関数が呼び出されるたびに**スタックフレーム**が作成され、以下の情報が格納される。

```
高位アドレス
+---------------------------+
|        関数の引数           |
+---------------------------+
|    戻りアドレス (RIP/EIP)   |  <-- call命令によって自動的に積まれる
+---------------------------+
|    保存されたベースポインタ   |  <-- 旧RBP/EBP
|         (SFP/RBP)         |
+---------------------------+
|                            |
|     ローカル変数            |  <-- バッファなど
|     (char buf[64] など)    |
|                            |
+---------------------------+
低位アドレス
```

ここで極めて重要なのは、**ローカル変数（バッファ）と戻りアドレスが同じスタック上に隣接して配置されている**という事実である。もしローカルバッファへの書き込みが境界を超えると、保存されたベースポインタや戻りアドレスを上書きしてしまう。攻撃者はこの性質を悪用して、戻りアドレスを任意の値に書き換え、プログラムの制御フローを乗っ取る。

## 3. スタックバッファオーバーフロー

### 3.1 脆弱なコードの例

以下は、典型的なスタックバッファオーバーフローの脆弱性を含むCプログラムである。

```c
#include <stdio.h>
#include <string.h>

void vulnerable_function(char *input) {
    char buffer[64]; // 64 bytes allocated on the stack
    strcpy(buffer, input); // No bounds checking!
    printf("You entered: %s\n", buffer);
}

int main(int argc, char *argv[]) {
    if (argc > 1) {
        vulnerable_function(argv[1]);
    }
    return 0;
}
```

`strcpy()`は、ソース文字列のヌル終端（`\0`）に達するまでコピーを続ける。入力がバッファサイズの64バイトを超えた場合、バッファの境界を越えて書き込みが行われる。

### 3.2 オーバーフロー時のスタックの変化

正常な状態とオーバーフロー発生時のスタックの変化を見てみよう。

```
【正常な状態】                      【オーバーフロー発生時】

高位アドレス                        高位アドレス
+------------------+              +------------------+
|   引数: *input   |              |   引数: *input   |
+------------------+              +------------------+
| 戻りアドレス       |              | 0x41414141       | <-- "AAAA"で上書き!
| (正規のアドレス)   |              | (攻撃者が制御)    |
+------------------+              +------------------+
| 保存された RBP    |              | 0x41414141       | <-- 上書き!
+------------------+              +------------------+
|                  |              | AAAAAAAAAAAAAAAA |
| buffer[64]       |              | AAAAAAAAAAAAAAAA |
| "Hello"          |              | AAAAAAAAAAAAAAAA | <-- 溢れた!
|                  |              | AAAAAAAAAAAAAAAA |
+------------------+              +------------------+
低位アドレス                        低位アドレス
```

バッファに64バイトを超えるデータ（例えば80バイトの`'A'`）を書き込むと、まずバッファの64バイトが埋まり、次に保存されたベースポインタ（8バイト、x86-64の場合）が上書きされ、さらにその上の戻りアドレスが上書きされる。

関数の`ret`命令が実行されると、スタックから戻りアドレスが読み取られ、その位置にジャンプする。戻りアドレスが`0x4141414141414141`（"AAAAAAAA"のASCIIコード）に書き換えられていると、プログラムは無効なアドレスにジャンプしようとしてクラッシュする（セグメンテーションフォルト）。

### 3.3 攻撃の原理：戻りアドレスの制御

単にクラッシュさせるだけでは攻撃にはならない。攻撃者の目的は、戻りアドレスを**意味のあるアドレス**に書き換えることである。攻撃者が戻りアドレスの位置を正確に把握し、そこに実行させたいコードのアドレスを配置できれば、プログラムの制御フローを完全に乗っ取ることができる。

```
攻撃ペイロードの構造:

+----------------------------------------------------------+
| NOP スレッド | シェルコード  |  パディング | 戻りアドレス     |
| (0x90...)   | (攻撃コード)  |  (調整用)  | (バッファ内を指す) |
+----------------------------------------------------------+
|<---------- buffer[64] ---------->|<- SFP ->|<-- RIP -->|
```

攻撃者はバッファ内にシェルコード（攻撃用の機械語命令列）を配置し、戻りアドレスをそのシェルコードの位置に書き換える。関数から`ret`命令で戻る際に、制御はシェルコードに移り、攻撃者が意図した処理（シェルの起動、ファイルの読み取り、ネットワーク接続の確立など）が実行される。

## 4. シェルコードの仕組み

### 4.1 シェルコードとは

シェルコードとは、exploit内で実行される小さな機械語（マシンコード）の断片である。名前は、歴史的に最も一般的な目的がUNIXシェル（`/bin/sh`）の起動であったことに由来する。実際には、任意の処理を行うコードであり得る。

シェルコードの基本的な要件は以下の通りである。

- **位置独立（Position-Independent）**: メモリ上のどのアドレスに配置されても正しく動作する必要がある
- **ヌルバイトフリー**: `strcpy()`などの文字列関数経由で注入する場合、ヌルバイト（`\0`）が含まれるとそこで文字列が終端されてしまうため、ヌルバイトを含まないように書く必要がある
- **小サイズ**: バッファのサイズに収まる必要がある

### 4.2 Linux x86-64におけるシェル起動シェルコードの概念

以下は、Linux x86-64で`/bin/sh`を起動するシェルコードのアセンブリの概念的な例である。

```asm
; execve("/bin/sh", NULL, NULL) syscall for Linux x86-64
; syscall number for execve: 59 (0x3b)

xor    rdi, rdi        ; Clear rdi
push   rdi             ; Push null terminator
mov    rdi, 0x68732f6e69622f  ; "/bin/sh" in little-endian
push   rdi             ; Push "/bin/sh" onto stack
mov    rdi, rsp        ; rdi = pointer to "/bin/sh"
xor    rsi, rsi        ; rsi = NULL (argv)
xor    rdx, rdx        ; rdx = NULL (envp)
mov    al, 0x3b        ; syscall number for execve
syscall                ; Execute the syscall
```

このコードは`execve`システムコールを直接呼び出してシェルを起動する。`xor reg, reg`を使ってレジスタをゼロクリアしているのは、`mov reg, 0`とすると命令のエンコーディングにヌルバイトが含まれるためである。

### 4.3 NOPスレッド（NOP Sled）

実際のexploitでは、シェルコードの正確なアドレスを特定することが困難な場合がある。そこで、NOPスレッド（NOP Sled）というテクニックが使われる。

NOP（No Operation、x86では`0x90`）命令は何も処理を行わず、次の命令に進むだけの命令である。シェルコードの前に大量のNOP命令を配置することで、ジャンプ先のアドレスが多少ずれても、NOPを順番に実行していき、最終的にシェルコードに到達する。

```
+-------------------------------------------------------+
| 0x90 0x90 0x90 ... 0x90 | shellcode | return address   |
| <--- NOP sled --------> |           |                  |
+-------------------------------------------------------+
       ^                        ^
       |                        |
  ジャンプ先が              ここに到達すれば
  この範囲内なら OK          攻撃成功
```

NOPスレッドは攻撃の「着地エリア」を広げる効果がある。ASLR（後述）が導入される以前は、この手法が非常に有効であった。

## 5. 具体的な脆弱コードとその修正

### 5.1 危険な関数群

C標準ライブラリには、境界チェックを行わない関数が多数存在する。これらは「使用禁止」とされるべき関数である。

| 危険な関数 | 安全な代替 | 問題点 |
|-----------|-----------|--------|
| `gets()` | `fgets()` | 入力長を一切チェックしない。C11で仕様から削除 |
| `strcpy()` | `strncpy()`, `strlcpy()` | コピー先バッファの境界をチェックしない |
| `strcat()` | `strncat()`, `strlcat()` | 結合先バッファの境界をチェックしない |
| `sprintf()` | `snprintf()` | 出力先バッファの境界をチェックしない |
| `scanf("%s", ...)` | `scanf("%63s", ...)` | 最大幅を指定しないと境界を超えて読み込む |

### 5.2 Stack Smashingの実例

以下に、より実践的な脆弱プログラムの例を示す。これは認証をバイパスするバッファオーバーフローの典型例である。

```c
#include <stdio.h>
#include <string.h>
#include <stdlib.h>

int check_password(char *input) {
    int authenticated = 0;       // Local variable (above buffer on stack)
    char password_buffer[16];    // 16-byte buffer on stack

    // Vulnerable: no bounds checking
    strcpy(password_buffer, input);

    if (strcmp(password_buffer, "secret123") == 0) {
        authenticated = 1;
    }

    return authenticated;
}

int main(int argc, char *argv[]) {
    if (argc < 2) {
        printf("Usage: %s <password>\n", argv[0]);
        return 1;
    }

    if (check_password(argv[1])) {
        printf("Access granted!\n");
        // ... privileged operations ...
    } else {
        printf("Access denied.\n");
    }

    return 0;
}
```

このプログラムでは、`password_buffer`の直後（高位アドレス側）に`authenticated`変数が配置される可能性がある。16バイトを超える入力を与えると、`authenticated`変数が上書きされ、0以外の値になることで認証をバイパスできる。

```
スタック上の配置（コンパイラ依存）:

+------------------+
| 戻りアドレス       |
+------------------+
| 保存されたRBP     |
+------------------+
| authenticated    |  <-- この値が0以外になれば認証通過
| (4 bytes)        |
+------------------+
| password_buffer  |  <-- ここから溢れると上の変数を上書き
| [16 bytes]       |
+------------------+
```

### 5.3 安全なコードへの修正

上記のコードを安全に書き直す方法を示す。

```c
#include <stdio.h>
#include <string.h>
#include <stdlib.h>

int check_password(const char *input) {
    char password_buffer[16];

    // Safe: use strncpy with explicit size limit
    strncpy(password_buffer, input, sizeof(password_buffer) - 1);
    password_buffer[sizeof(password_buffer) - 1] = '\0'; // Ensure null termination

    // Use constant-time comparison to prevent timing attacks
    if (strcmp(password_buffer, "secret123") == 0) {
        return 1;
    }

    return 0;
}
```

修正のポイントは以下の通りである。

- `strncpy()`を使い、コピーするバイト数を明示的に制限している
- バッファの最後にヌル終端を保証している（`strncpy()`はバッファが一杯の場合にヌル終端しない）
- `authenticated`変数をバッファと分離し、関数の戻り値として返している

## 6. ヒープオーバーフロー

### 6.1 ヒープの構造と脆弱性

スタックだけでなく、ヒープ上のバッファでもオーバーフローは発生する。ヒープオーバーフローはスタックオーバーフローほど直接的ではないが、同様に深刻な脆弱性である。

ヒープメモリは`malloc()`によって動的に確保される。多くのヒープ実装（glibc mallocなど）では、各メモリチャンクの前にメタデータ（チャンクヘッダー）が配置される。

```
+------------------+------------------+------------------+
| chunk header A   |   data A         | chunk header B   |   data B   ...
| (size, flags,    |   (user data)    | (size, flags,    |   (user data)
|  prev_size)      |                  |  prev_size)      |
+------------------+------------------+------------------+
```

### 6.2 ヒープオーバーフローの脆弱コード

```c
#include <stdio.h>
#include <stdlib.h>
#include <string.h>

typedef struct {
    char name[32];
    void (*callback)(void);  // Function pointer
} user_t;

void normal_action(void) {
    printf("Normal action executed.\n");
}

void admin_action(void) {
    printf("Admin action executed! (should not happen)\n");
    // ... dangerous privileged operation ...
}

int main(int argc, char *argv[]) {
    char *input_buffer = malloc(64);
    user_t *user = malloc(sizeof(user_t));

    user->callback = normal_action;
    strncpy(user->name, "user", sizeof(user->name) - 1);

    if (argc > 1) {
        // Vulnerable: overflow input_buffer into user struct
        strcpy(input_buffer, argv[1]);
    }

    printf("User: %s\n", user->name);
    user->callback();  // Call via function pointer

    free(input_buffer);
    free(user);
    return 0;
}
```

この例では、`input_buffer`と`user`構造体がヒープ上で隣接して配置される可能性がある。`input_buffer`に64バイトを超えるデータを書き込むと、隣接する`user`構造体のデータを上書きでき、特に`callback`関数ポインタを`admin_action`のアドレスに書き換えることで、意図しない特権操作を実行できる。

### 6.3 ヒープオーバーフローの悪用手法

ヒープオーバーフローの悪用は、スタックオーバーフローよりも複雑であり、ヒープ管理の実装に強く依存する。代表的な手法として以下がある。

**Unlink攻撃**: glibcの古い実装では、`free()`時にチャンクの連結リスト操作（unlink）が行われる。ヒープオーバーフローでチャンクヘッダーのポインタを書き換えると、unlink操作によって任意のアドレスに任意の値を書き込むことができた（Write-What-Where条件）。現在のglibcではこの攻撃に対する緩和策が実装されている。

**関数ポインタの上書き**: 上記の例のように、ヒープ上に関数ポインタを含む構造体がある場合、隣接するバッファからのオーバーフローで関数ポインタを書き換える。

**vtableポインタの上書き**: C++のオブジェクトがヒープに確保された場合、仮想関数テーブル（vtable）へのポインタを書き換えることで、仮想関数呼び出し時に攻撃者が用意した関数にジャンプさせる。

## 7. フォーマット文字列攻撃

### 7.1 フォーマット文字列の脆弱性

フォーマット文字列攻撃は、バッファオーバーフローとは異なるメモリ破壊攻撃であるが、同じく制御フローの乗っ取りにつながる重要な脆弱性クラスである。

```c
#include <stdio.h>

void vulnerable(char *user_input) {
    // VULNERABLE: user input used as format string
    printf(user_input);
}

void safe(char *user_input) {
    // SAFE: user input passed as argument
    printf("%s", user_input);
}
```

`printf(user_input)`と書くと、ユーザー入力がフォーマット文字列として解釈される。攻撃者が`%x %x %x %x`のような入力を与えると、スタック上のデータが16進数として出力され、メモリの内容が漏洩する（情報漏洩）。

### 7.2 `%n`による任意のメモリ書き込み

`printf`のフォーマット指定子`%n`は、「これまでに出力された文字数を、引数として渡されたポインタが指す位置に書き込む」という特殊な機能を持つ。

```c
int count;
printf("Hello%n", &count); // count = 5 ("Hello" is 5 characters)
```

攻撃者がフォーマット文字列を制御できる場合、`%n`を利用してスタック上のアドレスを参照させ、任意のメモリ位置に任意の値を書き込むことができる。これにより、GOT（Global Offset Table）のエントリを書き換えて制御フローを乗っ取るなどの攻撃が可能になる。

### 7.3 フォーマット文字列攻撃の防御

- ユーザー入力をフォーマット文字列として使用しない（常に`printf("%s", input)`の形式にする）
- コンパイラの警告を有効にする（`-Wformat -Wformat-security`）
- 最新のglibcでは`%n`の使用を制限するFORTIFY_SOURCEが有効になっている

## 8. 防御機構

バッファオーバーフロー攻撃の長い歴史の中で、多層的な防御機構が開発されてきた。現代のシステムでは、これらの防御が組み合わさって攻撃を困難にしている。

### 8.1 Stack Canary（スタックカナリア / SSP）

**原理**: 関数のプロローグでスタックのローカル変数と戻りアドレスの間にランダムな値（カナリア値）を挿入し、関数のエピローグでその値が変更されていないかをチェックする。バッファオーバーフローで戻りアドレスを上書きするためには、必ずカナリア値も上書きすることになるため、改ざんを検出できる。

名前の由来は、炭鉱でガス漏れを検知するために使われたカナリア（鳥）にちなんでいる。

```
カナリアが挿入されたスタック:

+------------------+
| 戻りアドレス       |
+------------------+
| 保存されたRBP     |
+------------------+
| Stack Canary     |  <-- ランダムな値。この値が変わったら
| (0xdeadbeef...)  |      オーバーフローが発生したと判断
+------------------+
|                  |
| ローカル変数      |  <-- バッファ。ここから溢れると
| (buffer[])       |      カナリアを破壊してしまう
|                  |
+------------------+
```

**GCCでの有効化**: `-fstack-protector`（一部の関数に適用）、`-fstack-protector-strong`（より多くの関数に適用）、`-fstack-protector-all`（すべての関数に適用）。現代のLinuxディストリビューションでは、デフォルトで`-fstack-protector-strong`が有効になっている。

**限界**: カナリアは線形的なオーバーフロー（バッファから順番に溢れるケース）を検出するが、任意のアドレスへの書き込み（Write-What-Where）やカナリアをスキップするようなオーバーフローには対応できない。また、カナリアの値がメモリリークなどで漏洩した場合、攻撃者はカナリアの値をそのまま維持しながらオーバーフローを行うことができる。

### 8.2 ASLR（Address Space Layout Randomization）

**原理**: プロセスのメモリレイアウト（スタック、ヒープ、共有ライブラリ、実行ファイルのベースアドレス）をプロセスの起動ごとにランダム化する。これにより、攻撃者がバッファのアドレスやライブラリ関数のアドレスを事前に予測することが困難になる。

```
通常（ASLRなし）:                  ASLR有効時:

libc: 0x7ffff7a00000            libc: 0x7f3a8c200000  (毎回変わる)
stack: 0x7ffffffde000           stack: 0x7ffc3a100000  (毎回変わる)
heap:  0x602000                 heap:  0x5618a3400000  (毎回変わる)
```

**Linuxでの設定**: `/proc/sys/kernel/randomize_va_space`で制御される。値`2`（デフォルト）で完全なランダム化が有効になる。

**限界**: ASLRのエントロピー（ランダム性のビット数）には限界がある。特に32ビットシステムではアドレス空間が小さく、ブルートフォース（繰り返しの試行）で正しいアドレスを推測できる場合がある。64ビットシステムではエントロピーが大幅に増加するが、情報漏洩（メモリアドレスの漏洩）があると無効化される。

### 8.3 DEP / NX bit（Data Execution Prevention / No-eXecute）

**原理**: メモリの各ページに「実行可能」か「書き込み可能」かの属性を設定し、書き込み可能な領域（スタック、ヒープ）でのコード実行を禁止する。これにより、攻撃者がスタックやヒープに注入したシェルコードの実行が防止される。W^X（Write XOR Execute）ポリシーとも呼ばれる。

```
DEP/NX有効時のメモリ属性:

Text セグメント:  r-x  (読み取り・実行可能、書き込み不可)
Data セグメント:  rw-  (読み書き可能、実行不可)
スタック:         rw-  (読み書き可能、実行不可)  <-- シェルコード実行不可!
ヒープ:          rw-  (読み書き可能、実行不可)  <-- シェルコード実行不可!
```

**ハードウェアサポート**: IntelのXD bit（eXecute Disable）、AMDのNX bit（No eXecute）として、CPU側でハードウェアレベルのサポートが提供されている。

**限界**: DEP/NXはコードの**注入**を防ぐが、既存のコード（ライブラリ内の命令列）を**再利用**する攻撃（ROP、ret2libcなど、後述）には対応できない。

### 8.4 PIE（Position-Independent Executable）

**原理**: 実行ファイル自体を位置独立なコードとしてコンパイルし、ASLRの効果を実行ファイルのTextセグメントにも適用する。PIEが無効な場合、実行ファイルのベースアドレスは固定されるため、実行ファイル内のROP gadget（後述）のアドレスが予測可能になってしまう。

**GCCでの有効化**: `-fpie -pie`フラグでコンパイル・リンクする。現代のLinuxディストリビューションでは多くの場合デフォルトで有効になっている。

### 8.5 RELRO（RELocation Read-Only）

**原理**: ELFバイナリの再配置（relocation）関連のセクションを読み取り専用にする。特にFull RELROでは、GOT（Global Offset Table）全体が読み取り専用に設定され、GOT上書き攻撃を防止する。

- **Partial RELRO**: `.got`セクションは読み取り専用だが、`.got.plt`は書き込み可能
- **Full RELRO**: 起動時にすべてのシンボル解決を完了し、GOT全体を読み取り専用にする

**GCCでの有効化**: `-Wl,-z,relro`（Partial）、`-Wl,-z,relro,-z,now`（Full）。

### 8.6 Control Flow Integrity（CFI）

**原理**: プログラムの制御フロー（関数呼び出しや分岐の遷移先）をコンパイル時に解析し、実行時に正当な遷移先のみを許可する。間接呼び出し（関数ポインタ経由の呼び出し）や`ret`命令の遷移先が、事前に定義された正当な値かどうかを検証する。

CFIには大きく分けて二つのカテゴリがある。

- **Forward-Edge CFI**: 間接的な関数呼び出し（`call`命令）の遷移先を検証する。関数ポインタの書き換えによる攻撃を防ぐ
- **Backward-Edge CFI**: `ret`命令の遷移先を検証する。ROP攻撃を防ぐ。IntelのCET（Control-flow Enforcement Technology）のShadow Stackがこれに該当する

**LLVM/Clangでの有効化**: `-fsanitize=cfi`フラグで有効化できる。

**限界**: CFIの精度はコンパイル時の解析能力に依存する。粗い粒度のCFIでは、正当な遷移先の集合が大きくなりすぎ、攻撃者がその中から都合の良い遷移先を選ぶことが可能な場合がある。

### 8.7 防御機構のまとめ

```
+-------------------+---------------------------+---------------------------+
|   防御機構         |   防御する攻撃              |   バイパス手法             |
+-------------------+---------------------------+---------------------------+
| Stack Canary      | スタックバッファ             | カナリア値の漏洩、          |
|                   | オーバーフロー               | 非線形書き込み             |
+-------------------+---------------------------+---------------------------+
| ASLR              | アドレスの推測による          | 情報漏洩、                 |
|                   | 攻撃                       | ブルートフォース（32bit）   |
+-------------------+---------------------------+---------------------------+
| DEP / NX          | スタック/ヒープ上の           | ROP, ret2libc,            |
|                   | コード実行                  | JOP                       |
+-------------------+---------------------------+---------------------------+
| PIE               | 実行ファイル内の             | 情報漏洩                   |
|                   | gadgetアドレスの推測         |                           |
+-------------------+---------------------------+---------------------------+
| RELRO             | GOT上書き攻撃              | Partial RELROでは          |
|                   |                           | .got.pltは書き換え可能     |
+-------------------+---------------------------+---------------------------+
| CFI               | 制御フロー                  | 粗い粒度のCFIでは          |
|                   | ハイジャック                 | 正当な遷移先を悪用         |
+-------------------+---------------------------+---------------------------+
```

## 9. 防御を回避する攻撃技術

防御機構の進化に伴い、攻撃技術も高度化してきた。現代のexploitは、複数の防御を同時に回避する洗練されたテクニックを組み合わせている。

### 9.1 Return-to-libc（ret2libc）

**原理**: DEP/NXによってスタック上のシェルコードが実行できなくなったことへの対抗策として、既にメモリ上にロードされているライブラリ（libc）内の関数を利用する手法。戻りアドレスを`system()`関数のアドレスに書き換え、引数に`"/bin/sh"`文字列のアドレスを設定することで、シェルコードを注入せずにシェルを起動する。

```
ret2libc攻撃のスタック構成（x86の場合）:

+------------------+
| "/bin/sh"のアドレス |  <-- system()の引数
+------------------+
| exit()のアドレス    |  <-- system()からの戻りアドレス
+------------------+
| system()のアドレス  |  <-- 元の戻りアドレスを上書き
+------------------+
| 保存されたRBP     |  <-- 上書き
+------------------+
| パディング         |
| (buffer[])       |
+------------------+
```

x86-64ではcalling conventionが異なり（引数がレジスタ渡し）、単純なret2libcではなくROPを組み合わせる必要がある。

### 9.2 ROP（Return-Oriented Programming）

**原理**: ROPは、既存のコード中の短い命令列（gadget）を連鎖させて、任意の処理を構築するチューリング完全な攻撃手法である。各gadgetは`ret`命令で終わる短い命令列であり、スタック上にgadgetのアドレスを連続して配置することで、`ret`命令を「次のgadgetへのジャンプ」として利用する。

```
ROP chain の実行フロー:

スタック:                        実行の流れ:
+------------------+
| gadget_3 のアドレス |  ----+
+------------------+     |
| gadget_2 のアドレス |  --+ |
+------------------+   | |
| gadget_1 のアドレス |  + | |   gadget_1:        gadget_2:        gadget_3:
+------------------+ | | |     pop rdi         pop rsi         syscall
| (buffer を埋める)   | | |     ret    -------> ret   -------->
+------------------+ | | |       |                |
                     v | |       v                v
               [実行開始]  [gadget_1が終了して    [gadget_2が終了して
                          gadget_2にret]        gadget_3にret]
```

例えば、`execve("/bin/sh", NULL, NULL)`を実行するROP chainは以下のようになる。

```
ROP chain for execve("/bin/sh", NULL, NULL) on x86-64:

+---------------------------+
| syscall gadget のアドレス    |  6. syscall を実行
+---------------------------+
| pop rax; ret のアドレス      |  5. rax = 59 (execve)
+---------------------------+
| 59 (0x3b)                 |     (pop rax の値)
+---------------------------+
| pop rdx; ret のアドレス      |  4. rdx = 0 (NULL)
+---------------------------+
| 0x0                       |     (pop rdx の値)
+---------------------------+
| pop rsi; ret のアドレス      |  3. rsi = 0 (NULL)
+---------------------------+
| 0x0                       |     (pop rsi の値)
+---------------------------+
| pop rdi; ret のアドレス      |  2. rdi = &"/bin/sh"
+---------------------------+
| "/bin/sh"のアドレス          |     (pop rdi の値)
+---------------------------+
| 元の戻りアドレス(上書き)      |  1. ここからROP chain開始
+---------------------------+
```

**ROPの強力さ**: ROPは既存の正当なコードの断片のみを使用するため、DEP/NXを完全に回避できる。また、必要なgadgetが見つかる限り、任意の処理を構築できるチューリング完全な攻撃手法である。

**gadgetの発見**: `ROPgadget`、`ropper`などのツールを使って、バイナリやライブラリ内からgadgetを自動的に検索できる。

### 9.3 JOP（Jump-Oriented Programming）

**原理**: ROPが`ret`命令を使用するのに対し、JOPは`jmp`命令や`call`命令で終わるgadgetを使用する。ROP対策としてのShadow Stack（`ret`命令の戻り先を別途記録・検証する仕組み）を回避するために開発された。

JOPでは、dispatcher gadget（次のgadgetのアドレスをレジスタから読み取ってジャンプする命令列）を中心にgadgetを連鎖させる。ROPほど広く使われてはいないが、`ret`命令を監視するBackward-Edge CFIに対する有効な回避策である。

### 9.4 情報漏洩によるASLRの無効化

ASLRはアドレスのランダム化によって攻撃を困難にするが、プログラムのバグによってメモリアドレスが漏洩すると、ランダム化の効果は失われる。

代表的な情報漏洩の手法は以下の通りである。

- **フォーマット文字列脆弱性**: `%p`や`%x`でスタック上のアドレス値を読み取る
- **部分的オーバーフロー**: バッファオーバーフローでヌル終端を上書きし、隣接するポインタ値を`printf()`等で出力させる
- **Use-After-Free**: 解放されたメモリ上のポインタを読み取る
- **サイドチャネル攻撃**: キャッシュタイミングなどを利用してアドレスを推測する

現実のexploitでは、まず情報漏洩の脆弱性でASLRを無効化し、次にバッファオーバーフローでROP chainを実行するという二段階の攻撃が一般的である。

## 10. 歴史的なインシデント

### 10.1 Morris Worm（1988年）

インターネット上で初めて大規模に拡散したワームである。Cornell大学の大学院生Robert Tappan Morrisが作成し、1988年11月2日にリリースされた。

Morris Wormは複数の脆弱性を悪用したが、その一つがUNIXの`fingerd`プログラムにおけるスタックバッファオーバーフローであった。`fingerd`は`gets()`関数を使用してネットワークからの入力を読み取っており、512バイトのバッファに対して境界チェックが行われていなかった。

このワームは当時のインターネット接続コンピュータの約10%（約6,000台）に感染し、多くのシステムをクラッシュさせた。この事件は、CERTコーディネーションセンターの設立のきっかけとなり、コンピュータセキュリティという分野の重要性を世界に認識させた。Morrisは、コンピュータ詐欺と不正利用に関する法律（Computer Fraud and Abuse Act）で起訴された最初の人物となった。

### 10.2 Code Red（2001年）

2001年7月に発見されたワームで、Microsoft IIS（Internet Information Services）のWebサーバーに含まれるバッファオーバーフロー脆弱性（MS01-033）を悪用した。具体的には、IISのIndex Serverに対するHTTPリクエストの処理における`.ida`ファイルの拡張子マッピングにバッファオーバーフローが存在した。

Code Redは約36万台のサーバーに感染し、ホワイトハウスのWebサイトに対するDDoS攻撃を試みた。感染速度は驚異的で、発見から14時間以内に25万台以上に拡散した。このワームによる経済的損失は約26億ドルと推定されている。

### 10.3 SQL Slammer（2003年）

2003年1月に発見されたワームで、Microsoft SQL Server 2000のSQL Server Resolution Serviceにおけるスタックバッファオーバーフロー脆弱性（MS02-039）を悪用した。

SQL Slammerの特筆すべき点はその拡散速度である。わずか376バイトのUDPパケット一つがexploitのペイロード全体であり、最初の感染から約10分で脆弱なホストの90%に拡散した。この速度は、理論上の最大拡散速度に近いものであった。Bank of Americaの13,000台のATM、韓国のインターネットインフラ、米国の原子力発電所の監視システムなど、社会的に重要なシステムに広範な影響を与えた。

### 10.4 Heartbleed（2014年）

厳密にはバッファ「オーバーフロー」ではなくバッファ「オーバーリード」（境界を超えたメモリ読み取り）であるが、メモリ安全性の欠如に起因する脆弱性として極めて重要な事例である。OpenSSLのTLS Heartbeat拡張の実装において、クライアントが指定した長さフィールドを検証せずにメモリの内容を返送するバグがあり、サーバーのメモリ上の秘密鍵やセッション情報が漏洩する可能性があった。

CVE-2014-0160として登録され、当時のインターネット上のHTTPSサーバーの約17%（約50万台）が影響を受けたと推定される。

### 10.5 EternalBlue（2017年）

米国国家安全保障局（NSA）が開発したとされるexploitで、Windowsの SMB（Server Message Block）プロトコルの実装におけるバッファオーバーフロー脆弱性（MS17-010）を悪用する。2017年にShadow Brokersグループによって公開され、WannaCryランサムウェアやNotPetyaの拡散に使用された。WannaCryは150カ国以上、23万台以上のコンピュータに感染し、英国のNHS（国民保健サービス）を含む多くの組織に深刻な影響を与えた。

## 11. メモリ安全な言語による根本的解決

### 11.1 問題の本質

これまで見てきた防御機構は、すべて「C/C++のメモリ安全性の欠如」という根本的な問題に対する**緩和策**（mitigation）である。攻撃を困難にすることはできるが、根本的にバッファオーバーフローの発生を防ぐものではない。真の解決策は、メモリ安全な言語を使用してソフトウェアを開発することである。

### 11.2 Rust

Rustは、メモリ安全性をコンパイル時に保証する言語として2015年に安定版がリリースされた。所有権（Ownership）システムと借用チェッカー（Borrow Checker）によって、以下のメモリ安全性の問題をコンパイル時に検出・防止する。

- バッファオーバーフロー: 配列アクセスに自動的に境界チェックが挿入される
- Use-After-Free: 所有権システムにより、解放されたメモリへのアクセスがコンパイルエラーになる
- 二重解放: 所有権が一つの変数にのみ属するため、同じメモリの二重解放が構造的に不可能
- データ競合: 借用規則により、可変参照の排他性がコンパイル時に保証される

```rust
fn safe_example() {
    let buffer = vec![0u8; 64];
    // Compile-time known size: direct indexing is bounds-checked at runtime
    // buffer[100] would panic at runtime with "index out of bounds"

    // Idiomatic Rust uses iterators and slices, avoiding indexing altogether
    let input = "Hello, World!";
    let bytes = input.as_bytes();

    // Safe: get() returns Option<&T>, no out-of-bounds access possible
    match buffer.get(100) {
        Some(val) => println!("Value: {}", val),
        None => println!("Index out of bounds - handled safely"),
    }
}
```

Rustの重要な特徴は、メモリ安全性を実現しつつ、C/C++と同等のパフォーマンスを達成することである。ガベージコレクタを使用しないため、リアルタイム性が求められるシステムプログラミングにも適用できる。

Linux カーネルは2022年からRustを第二のプログラミング言語として公式にサポートし始めた。Android、Windows、Chromiumなどの大規模プロジェクトでもRustの採用が進んでいる。

### 11.3 Go

Goはガベージコレクションを備えた言語であり、手動のメモリ管理に起因するバッファオーバーフローやUse-After-Freeの問題を回避する。配列・スライスのアクセスには自動的に境界チェックが挿入され、範囲外アクセスはpanicとなる。

```go
func safeExample() {
    buffer := make([]byte, 64)
    // buffer[100] = 0  // This would panic: "index out of range [100] with length 64"

    // Safe pattern: check length before access
    index := 100
    if index < len(buffer) {
        buffer[index] = 0
    }
}
```

GoはCと比較してパフォーマンスのオーバーヘッドがあるが（ガベージコレクションの停止時間、境界チェックのコスト）、Webサービス、CLIツール、ネットワークプログラミングなど多くの用途で十分なパフォーマンスを提供する。

### 11.4 メモリ安全な言語の採用状況

2022年のNSA（米国国家安全保障局）の勧告「Software Memory Safety」では、メモリ安全な言語への移行が推奨された。2023年にはCISA（Cybersecurity and Infrastructure Security Agency）も同様の勧告を発出し、2024年のホワイトハウス技術報告書「Back to the Building Blocks」でも、ソフトウェア業界全体がメモリ安全な言語に移行すべきだという明確なメッセージが示された。

GoogleはAndroidにおける新規コードのメモリ安全な言語での記述を推進し、その結果としてメモリ安全性の脆弱性が2019年の76%から2024年には24%に減少したと報告している。MicrosoftはWindowsの脆弱性の約70%がメモリ安全性の問題であると公表しており、Rustの採用を積極的に進めている。

## 12. 現代における関連性

### 12.1 バッファオーバーフローは過去の問題ではない

防御機構の発達とメモリ安全な言語の普及にもかかわらず、バッファオーバーフローは現代でも重大な脅威であり続けている。その理由は以下の通りである。

**膨大なレガシーコード**: オペレーティングシステムのカーネル、ネットワークプロトコルスタック、組み込みシステムのファームウェア、暗号ライブラリなど、社会インフラの基盤を成すソフトウェアの多くがC/C++で書かれている。これらを一夜にしてメモリ安全な言語に書き換えることは不可能である。

**組み込みシステムとIoT**: リソースが制約された組み込みシステムでは、依然としてCが主要な開発言語である。IoT（Internet of Things）デバイスの急速な普及に伴い、攻撃対象となるCコードベースは増加している。

**高度な攻撃技術の進化**: 前述のとおり、ROP、JOP、情報漏洩を組み合わせた攻撃技術は年々高度化しており、既存の防御機構を回避する手法が継続的に研究されている。

### 12.2 現代のexploit開発

現代のバッファオーバーフローexploitは、2000年代初頭のような単純なスタック書き換えとは異なり、以下のような多段階の攻撃チェーンとなっている。

1. **情報漏洩**: フォーマット文字列脆弱性やバッファオーバーリードを使って、メモリ上のアドレス情報を取得する。これによりASLRを無効化する
2. **Stack Canaryのバイパス**: 情報漏洩でカナリア値を取得するか、カナリアに影響しないオーバーフロー経路を利用する
3. **ROP chainの構築**: 漏洩したアドレス情報をもとに、正確なgadgetアドレスを計算してROP chainを構築する
4. **ペイロードの実行**: ROP chainを使って`mprotect()`などでメモリの実行権限を変更し、より大きなペイロード（シェルコード）を実行する

このような高度な攻撃に対しては、前述のCFI、Shadow Stack（Intel CET）、Memory Tagging Extension（ARM MTE）などの新しいハードウェアベースの防御機構が研究・実装されている。

### 12.3 今後の展望

メモリ安全性の問題は、コンピュータセキュリティにおいて依然として最大の課題の一つである。今後の方向性として以下が注目される。

**ハードウェアによる防御の強化**: Intel CET（Shadow StackとIndirect Branch Tracking）、ARM MTE（Memory Tagging Extension）、CHERI（Capability Hardware Enhanced RISC Instructions）など、ハードウェアレベルでのメモリ安全性の保証が進んでいる。特にCHERIは、ポインタにメタデータ（ケイパビリティ）を持たせることで、きめ細かいメモリアクセス制御を実現する野心的なプロジェクトである。

**段階的な言語移行**: 既存のC/C++コードベースを一括でRustに書き換えることは非現実的であるため、モジュール単位での段階的な移行が現実的なアプローチとなっている。C/RustのFFI（Foreign Function Interface）による相互運用が重要な技術基盤である。

**形式検証とファジング**: コードの正当性を数学的に証明する形式検証（Formal Verification）と、大量のランダム入力によって脆弱性を発見するファジング（Fuzzing）が、ソフトウェア開発プロセスに組み込まれるようになっている。GoogleのOSS-Fuzzは、主要なオープンソースプロジェクトに対して継続的にファジングを実施し、多数のバッファオーバーフロー脆弱性を発見している。

## 13. まとめ

バッファオーバーフローは、C/C++のメモリ安全性の欠如に起因する根本的な脆弱性クラスである。その本質は、「プログラムが確保したメモリの境界を超えてデータを書き込み、隣接するメモリ上の重要なデータ（戻りアドレス、関数ポインタ、制御構造）を上書きすることで、プログラムの制御フローを乗っ取る」ことにある。

1988年のMorris Wormから2020年代のEternalBlueに至るまで、バッファオーバーフローは繰り返し壊滅的なセキュリティインシデントの原因となってきた。これに対し、Stack Canary、ASLR、DEP/NX、PIE、RELRO、CFIといった多層的な防御機構が開発されてきたが、これらはいずれも攻撃を**困難に**するものであって、**不可能に**するものではない。ROP、ret2libc、JOPなどの高度な攻撃技術は、これらの防御を回避する方法を示してきた。

真の解決策は、メモリ安全な言語（Rust、Goなど）を使用してソフトウェアを開発することである。業界全体がこの方向に向かいつつあるが、膨大なレガシーコードの存在と、組み込みシステムにおけるCの支配的地位を考えると、バッファオーバーフローはこれからも長期にわたってセキュリティ上の重要課題であり続けるだろう。

ソフトウェア開発者にとって重要なのは、バッファオーバーフローの原理を深く理解し、安全なコーディング実践を身につけ、利用可能な防御機構を最大限に活用し、可能であればメモリ安全な言語を選択することである。セキュリティは単一の技術で解決できる問題ではなく、多層的な防御の積み重ねによって実現されるものである。
