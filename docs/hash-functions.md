# 暗号学的ハッシュ関数（SHA-2/SHA-3）— 衝突耐性とMerkle-Damgård構造

## 1. はじめに：ハッシュ関数とは何か

**ハッシュ関数**は、任意長の入力データを固定長の出力値（ハッシュ値、ダイジェスト）に変換する関数である。暗号学的ハッシュ関数は、暗号プロトコルの構成要素として不可欠な存在であり、デジタル署名、メッセージ認証、パスワード保存、データの完全性検証など、セキュリティの幅広い分野で使われている。

$$
H: \{0, 1\}^* \rightarrow \{0, 1\}^n
$$

任意長のビット列を固定長 $n$ ビットのハッシュ値に写像する。例えばSHA-256では $n = 256$ である。

ハッシュ関数は暗号技術の「ユーティリティ関数」とも言える存在であり、ほぼすべての暗号プロトコルがその安全性の一部をハッシュ関数に依拠している。

## 2. 暗号学的ハッシュ関数の安全性要件

暗号学的ハッシュ関数は、以下の3つの性質を満たす必要がある：

### 2.1 原像耐性（Preimage Resistance）

ハッシュ値 $h$ が与えられたとき、$H(m) = h$ を満たす $m$ を見つけることが計算量的に困難であること。

$$
\text{Given } h, \text{ find } m \text{ such that } H(m) = h
$$

直感的には「ハッシュ値から元のメッセージを逆算できない」という性質である。$n$ ビットのハッシュ関数に対する最良の攻撃は $O(2^n)$ の計算量が期待される。

### 2.2 第二原像耐性（Second Preimage Resistance）

メッセージ $m_1$ が与えられたとき、$m_1 \neq m_2$ かつ $H(m_1) = H(m_2)$ を満たす $m_2$ を見つけることが計算量的に困難であること。

$$
\text{Given } m_1, \text{ find } m_2 \neq m_1 \text{ such that } H(m_1) = H(m_2)
$$

「同じハッシュ値を持つ別のメッセージを意図的に作成できない」という性質であり、署名の偽造防止に不可欠である。

### 2.3 衝突耐性（Collision Resistance）

$H(m_1) = H(m_2)$ かつ $m_1 \neq m_2$ を満たす任意の $(m_1, m_2)$ のペアを見つけることが計算量的に困難であること。

$$
\text{Find } m_1 \neq m_2 \text{ such that } H(m_1) = H(m_2)
$$

**誕生日のパラドックス**により、$n$ ビットのハッシュ関数に対する衝突探索の期待計算量は $O(2^{n/2})$ である。これは、約 $2^{n/2}$ 個のメッセージをハッシュすれば、50%以上の確率で衝突が見つかるという確率論的な性質に基づく。

したがって、128ビットの衝突耐性を持つためには256ビット以上のハッシュ出力が必要である。これがSHA-256が広く使われる理由の一つである。

## 3. Merkle-Damgård構造

### 3.1 基本的な考え方

多くのハッシュ関数（MD5、SHA-1、SHA-2）は**Merkle-Damgård構造**に基づいている。この構造は、固定長の入力を処理する**圧縮関数**を繰り返し適用することで、任意長の入力を処理する。

$$
f: \{0, 1\}^n \times \{0, 1\}^b \rightarrow \{0, 1\}^n
$$

圧縮関数 $f$ は、$n$ ビットの中間状態と $b$ ビットのメッセージブロックを受け取り、$n$ ビットの新しい中間状態を出力する。

```
Message:    M₁        M₂        M₃       M₄||padding||length
            │          │          │          │
            ↓          ↓          ↓          ↓
IV ──→ [f] ──→ h₁ ──→ [f] ──→ h₂ ──→ [f] ──→ h₃ ──→ [f] ──→ Hash
```

1. メッセージをブロックに分割する
2. 初期値 **IV**（Initial Value）から開始する
3. 各ブロックに対して圧縮関数を適用し、中間状態を更新する
4. 最後のブロックにはメッセージ長のパディングを追加する（**Merkle-Damgård強化**）
5. 最終的な中間状態がハッシュ値となる

### 3.2 安全性

Merkle-Damgård構造の重要な性質として、**圧縮関数が衝突耐性を持つならば、ハッシュ関数全体も衝突耐性を持つ**ことが証明されている。これにより、ハッシュ関数の安全性分析を圧縮関数の安全性分析に帰着させることができる。

### 3.3 Length Extension Attack

Merkle-Damgård構造には**長さ拡張攻撃（Length Extension Attack）**という既知の弱点がある。

$H(m)$ と $|m|$（メッセージ長）を知っている攻撃者は、$m$ を知らなくても、任意のメッセージ $m'$ に対して $H(m \| \text{pad}(m) \| m')$ を計算できる。これは最終的な内部状態がハッシュ値としてそのまま出力されるために生じる。

この攻撃は、$H(\text{secret} \| \text{message})$ をMAC（メッセージ認証コード）として使用するナイーブな構成を破る。対策として：

- **HMAC**を使用する（後の記事で詳述）
- SHA-3（Keccak）はスポンジ構造を使用しており、この攻撃を受けない

## 4. SHA-2ファミリー

### 4.1 歴史

SHA-2はNSA（米国国家安全保障局）が設計し、2001年にNISTが標準化したハッシュ関数ファミリーである。

前身であるSHA-1（160ビット出力）は、2005年にWangらによって理論的な衝突攻撃が発表され、2017年にGoogleとCWIアムステルダムが実際の衝突を生成した（SHAttered攻撃）。これを受けてSHA-1は非推奨となり、SHA-2への移行が進められた。

### 4.2 SHA-2のバリエーション

| アルゴリズム | 出力長 | 内部状態 | ブロック長 | ラウンド数 |
|---|---|---|---|---|
| SHA-224 | 224ビット | 256ビット | 512ビット | 64 |
| SHA-256 | 256ビット | 256ビット | 512ビット | 64 |
| SHA-384 | 384ビット | 512ビット | 1024ビット | 80 |
| SHA-512 | 512ビット | 512ビット | 1024ビット | 80 |
| SHA-512/224 | 224ビット | 512ビット | 1024ビット | 80 |
| SHA-512/256 | 256ビット | 512ビット | 1024ビット | 80 |

SHA-256とSHA-512が最も広く使われている。SHA-512は64ビットCPUでSHA-256より高速に動作するため、64ビット環境ではSHA-512/256が推奨されることもある。

### 4.3 SHA-256の内部構造

SHA-256の圧縮関数は以下の要素で構成される：

**初期ハッシュ値（IV）**：最初の8つの素数の平方根の小数部分から導出された8つの32ビット定数：

$$
H_0^{(0)} = \texttt{6a09e667}, \quad H_1^{(0)} = \texttt{bb67ae85}, \quad \ldots
$$

**ラウンド定数**：最初の64個の素数の立方根の小数部分から導出された64個の32ビット定数 $K_0, K_1, \ldots, K_{63}$。

**メッセージスケジュール**：512ビットのメッセージブロックを64個の32ビットワードに拡張する：

- $W_t = M_t$ （$0 \leq t \leq 15$、メッセージブロックをそのまま使用）
- $W_t = \sigma_1(W_{t-2}) + W_{t-7} + \sigma_0(W_{t-15}) + W_{t-16}$ （$16 \leq t \leq 63$）

ここで $\sigma_0$ と $\sigma_1$ はビット回転とシフトの組み合わせである：

$$
\sigma_0(x) = \text{ROTR}^7(x) \oplus \text{ROTR}^{18}(x) \oplus \text{SHR}^3(x)
$$
$$
\sigma_1(x) = \text{ROTR}^{17}(x) \oplus \text{ROTR}^{19}(x) \oplus \text{SHR}^{10}(x)
$$

**ラウンド処理**：8つの作業変数 $a, b, c, d, e, f, g, h$ を使用し、64ラウンドの処理を行う。各ラウンドでは以下の計算を行う：

$$
T_1 = h + \Sigma_1(e) + \text{Ch}(e, f, g) + K_t + W_t
$$
$$
T_2 = \Sigma_0(a) + \text{Maj}(a, b, c)
$$

ここで：
$$
\text{Ch}(e, f, g) = (e \wedge f) \oplus (\neg e \wedge g) \quad \text{（選択関数）}
$$
$$
\text{Maj}(a, b, c) = (a \wedge b) \oplus (a \wedge c) \oplus (b \wedge c) \quad \text{（多数決関数）}
$$
$$
\Sigma_0(a) = \text{ROTR}^2(a) \oplus \text{ROTR}^{13}(a) \oplus \text{ROTR}^{22}(a)
$$
$$
\Sigma_1(e) = \text{ROTR}^6(e) \oplus \text{ROTR}^{11}(e) \oplus \text{ROTR}^{25}(e)
$$

各ラウンドの最後に作業変数をシフトする：

$$
h \leftarrow g, \quad g \leftarrow f, \quad f \leftarrow e, \quad e \leftarrow d + T_1
$$
$$
d \leftarrow c, \quad c \leftarrow b, \quad b \leftarrow a, \quad a \leftarrow T_1 + T_2
$$

64ラウンド後、各作業変数を前のブロックの中間ハッシュ値に加算して、新しい中間ハッシュ値を得る。

### 4.4 SHA-2の安全性

SHA-2は登場から20年以上経過しているが、実用的な攻撃は発見されていない。

- SHA-256に対する最良の衝突攻撃は31ラウンドまで（全64ラウンド中）
- 原像攻撃は45ラウンドまで
- 全ラウンドに対する攻撃は総当たりより本質的に速い方法が知られていない

SHA-2は安全であると広く信頼されているが、SHA-1の教訓から、代替手段（SHA-3）を準備しておくことの重要性が認識された。

## 5. SHA-3（Keccak）

### 5.1 背景と選定

SHA-1の脆弱性が明らかになった後、NISTは2007年にSHA-3の公募を開始した。5年間にわたる評価プロセスを経て、2012年にGuido Bertoni、Joan Daemen（AESの共同設計者）、Michaël Peeters、Gilles Van Asscheが設計した**Keccak**がSHA-3として選ばれた。

SHA-3が選ばれた主な理由は、SHA-2とは根本的に異なる設計であることだった。SHA-2がMerkle-Damgård構造を使うのに対し、KeccakはShannonの混乱と拡散の原理を直接的に実現する**スポンジ構造**を採用している。

### 5.2 スポンジ構造

スポンジ構造は、Merkle-Damgård構造とは根本的に異なるハッシュ関数の構成法である。

```
                   r bits    c bits
                  ┌───────┬────────┐
                  │       │        │
Input ──→ XOR ──→│   f   │        │──→ XOR ──→ ... ──→ Output
  M₁       │     │       │        │      │      M₂
           │     └───────┴────────┘     │
           │        Absorb Phase        │
           └────────────────────────────┘
                                         Squeeze Phase
```

スポンジ構造は以下の要素で構成される：

- **状態（State）**：$b = r + c$ ビットの内部状態
- **レート（Rate）$r$**：各ステップで入出力されるビット数
- **キャパシティ（Capacity）$c$**：安全性パラメータ（外部に直接露出しない）
- **置換関数 $f$**：$b$ ビットの全単射（鍵のない置換）

処理は2つのフェーズからなる：

**吸収フェーズ（Absorbing）**：
1. メッセージを $r$ ビットのブロックに分割する
2. 各ブロックを状態のレート部分にXORする
3. 置換関数 $f$ を適用する

**搾出フェーズ（Squeezing）**：
1. 状態のレート部分 $r$ ビットを出力する
2. 必要に応じて $f$ を適用してさらに出力する

この構造の安全性レベルは $c/2$ ビットである。例えばSHA3-256では $c = 512$ であり、$256$ ビットの安全性を持つ。

### 5.3 Keccak-f置換

Keccakの状態は $5 \times 5 \times w$ ビットの3次元配列として表される。SHA-3では $w = 64$ であり、状態サイズは $b = 5 \times 5 \times 64 = 1600$ ビットである。

Keccak-f[1600]は24ラウンドの処理で構成され、各ラウンドは以下の5つのステップからなる：

- **$\theta$（シータ）**：列方向のパリティ計算とXOR拡散
- **$\rho$（ロー）**：ビット回転
- **$\pi$（パイ）**：レーンの転置
- **$\chi$（カイ）**：非線形変換（唯一の非線形ステップ）
- **$\iota$（イオタ）**：ラウンド定数の加算

$\chi$ の非線形変換は以下の式で表される：

$$
a'[x][y] = a[x][y] \oplus (\neg a[x+1][y] \wedge a[x+2][y])
$$

この式は3ビット入力1ビット出力のS-Boxに相当し、AESのS-Boxよりもシンプルだが、24ラウンドの繰り返しにより十分な安全性を実現している。

### 5.4 SHA-3のバリエーション

| アルゴリズム | 出力長 | $r$ | $c$ | 安全性レベル |
|---|---|---|---|---|
| SHA3-224 | 224 | 1152 | 448 | 112ビット |
| SHA3-256 | 256 | 1088 | 512 | 128ビット |
| SHA3-384 | 384 | 832 | 768 | 192ビット |
| SHA3-512 | 512 | 576 | 1024 | 256ビット |
| SHAKE128 | 可変 | 1344 | 256 | 128ビット |
| SHAKE256 | 可変 | 1088 | 512 | 256ビット |

**SHAKE128/SHAKE256**は**拡張出力関数（XOF: Extendable Output Function）**であり、任意長のハッシュ値を生成できる。鍵導出やストリーム暗号的な用途に使用される。

### 5.5 SHA-3の利点

- **長さ拡張攻撃への耐性**：スポンジ構造の設計により、SHA-2のような長さ拡張攻撃が原理的に不可能
- **設計の多様性**：SHA-2とは完全に異なるアプローチであり、SHA-2に万が一の脆弱性が見つかった場合のフォールバックとして機能する
- **柔軟性**：XOFにより可変長出力が可能
- **ハードウェア実装の効率性**：単純なビット演算のみで構成されており、FPGAやASICでの実装に適している

## 6. SHA-2とSHA-3の比較

| 特性 | SHA-2 | SHA-3（Keccak） |
|---|---|---|
| 構造 | Merkle-Damgård | スポンジ構造 |
| 長さ拡張攻撃 | 脆弱 | 耐性あり |
| ソフトウェア性能 | 非常に高速 | SHA-2よりやや遅い |
| ハードウェア性能 | 高速 | 非常に高速 |
| 設計年 | 2001 | 2012 |
| 実績 | 20年以上の実績 | 比較的新しい |
| ハードウェア支援 | SHA-NI命令 | 一部CPU |

現時点ではSHA-2（特にSHA-256）が最も広く使われているが、新しいプロトコルではSHA-3の採用も増えている。両者は互いの代替として共存する関係にある。

## 7. 実世界での利用

### 7.1 デジタル署名

RSA-PSS、ECDSA、EdDSAなどの署名アルゴリズムは、署名前にメッセージをハッシュする。ハッシュ関数の衝突耐性が損なわれると、異なるメッセージに対して同じ署名が有効になり、署名の安全性が崩壊する。

### 7.2 TLS/HTTPS

TLS 1.3ではSHA-256とSHA-384が暗号スイートに組み込まれている。ハンドシェイクのトランスクリプトハッシュ、鍵導出（HKDF）、完了メッセージの検証などに使われる。

### 7.3 ブロックチェーン

Bitcoinではブロックヘッダのハッシュ計算にSHA-256を2回適用（double SHA-256）する。マイニングはこのハッシュ値が特定の条件（先頭の0ビット数）を満たすnonce値を見つける作業である。

Ethereumでは独自のKeccak-256（SHA-3の最終標準化前のバージョン）を使用している。

### 7.4 Gitのオブジェクト管理

Gitはファイルやコミットの識別にSHA-1ハッシュを使用している。SHA-1の衝突が実証されたことを受けて、SHA-256への移行が進められている。

### 7.5 ファイルの完全性検証

ソフトウェアのダウンロードでは、SHA-256ハッシュ値をチェックサムとして公開し、ダウンロードしたファイルの改ざんや破損を検出する。

## 8. ハッシュ関数の誤用

### 8.1 パスワード保存にSHA-256を直接使わない

SHA-256は高速に設計されているため、パスワードのハッシュには適さない。攻撃者はGPUを使って毎秒数十億回のハッシュ計算が可能であり、辞書攻撃やブルートフォースに対して脆弱である。

パスワードの保存には、意図的に遅い**鍵導出関数**（bcrypt、Argon2など）を使用すべきである。

### 8.2 MACとしてのナイーブな使用

$H(\text{key} \| \text{message})$ をMACとして使用するのは、長さ拡張攻撃により安全でない。代わりにHMACを使用する。

### 8.3 ハッシュの切り詰め

256ビットのハッシュ値を128ビットに切り詰めると、衝突耐性は $2^{64}$ に低下する。必要な安全性レベルに応じた出力長のハッシュ関数を選択すべきである。

## 9. まとめ

暗号学的ハッシュ関数は、現代の暗号システムを支える最も基本的な構成要素の一つである。

- **SHA-2**（特にSHA-256）は20年以上の実績を持ち、現在も最も広く使われている
- **SHA-3**はSHA-2とは異なるスポンジ構造を採用し、長さ拡張攻撃への耐性など追加の安全性を提供する
- 両者は互いのバックアップとして共存しており、SHA-2に問題が見つかった場合にSHA-3に迅速に移行できる体制が整っている

ハッシュ関数の選択基準：

- 一般的な用途には**SHA-256**が標準
- 長さ拡張攻撃への耐性が必要な場合は**SHA-3**またはHMACを使用
- パスワード保存には**ハッシュ関数を直接使わず**、専用の鍵導出関数を使用
- 可変長出力が必要な場合は**SHAKE128/SHAKE256**を使用

ハッシュ関数は「暗号の接着剤」とも呼ばれ、一見地味ながら、暗号システム全体の安全性を根底で支える重要な技術である。
