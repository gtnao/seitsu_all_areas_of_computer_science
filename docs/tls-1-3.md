# TLS 1.3ハンドシェイク — 0-RTT、前方秘匿性、そしてTLS 1.2からの進化

## 1. はじめに：TLSとは

**TLS（Transport Layer Security）**は、インターネット上の通信を暗号化するプロトコルである。Webブラウザでの HTTPS 通信、メール送信（SMTPS）、VPN接続など、事実上すべての安全な通信がTLSに依存している。

TLSは**ハンドシェイク**と呼ばれる初期交渉フェーズで暗号パラメータを確立し、その後**レコードプロトコル**でデータを暗号化して送受信する。

2018年にRFC 8446として標準化された**TLS 1.3**は、TLS 1.2からの大幅な改良であり、セキュリティの強化、ハンドシェイクの高速化、プロトコルの簡素化を同時に実現した。

## 2. TLS 1.2の問題点

TLS 1.3の設計を理解するために、TLS 1.2の主な問題点を確認する。

### 2.1 ハンドシェイクの遅延

TLS 1.2のフルハンドシェイクは**2-RTT（Round Trip Time）**を要する：

```
Client                                    Server
──────                                    ──────
ClientHello          ──────────→
                     ←──────────          ServerHello
                                          Certificate
                                          ServerKeyExchange
                                          ServerHelloDone
ClientKeyExchange    ──────────→
ChangeCipherSpec
Finished             ──────────→
                     ←──────────          ChangeCipherSpec
                                          Finished
[Application Data]   ←────────→          [Application Data]
```

この2-RTTのオーバーヘッドは、特にモバイルネットワークや高レイテンシ環境で大きな問題となる。

### 2.2 レガシー暗号の残存

TLS 1.2では以下のような安全でない暗号スイートがサポートされていた：

- **RSA鍵交換**：前方秘匿性がない
- **CBC暗号モード**：パディングオラクル攻撃（Lucky Thirteen、POODLE）
- **RC4**：多数の統計的偏りが発見
- **SHA-1**：衝突が実証済み
- **静的DH**：前方秘匿性がない

### 2.3 複雑な暗号スイート交渉

TLS 1.2では暗号スイートが「鍵交換_署名_暗号_ハッシュ」の組み合わせで定義されており、数百の組み合わせが存在した。この複雑さがダウングレード攻撃やミスコンフィギュレーションの温床となった。

### 2.4 平文のメタデータ

TLS 1.2ではハンドシェイクメッセージの多く（Certificate、ServerKeyExchangeなど）が平文で送信されており、サーバーの身元が盗聴者に露出していた。

## 3. TLS 1.3の設計原則

TLS 1.3は以下の原則に基づいて設計された：

1. **前方秘匿性の強制**：すべての鍵交換をEphemeral DHで行う
2. **ハンドシェイクの高速化**：1-RTT（および0-RTT）のハンドシェイク
3. **レガシーの排除**：安全でない暗号の完全廃止
4. **暗号化の拡大**：ハンドシェイクメッセージの大部分を暗号化
5. **シンプルさ**：暗号スイートの大幅な簡素化

## 4. TLS 1.3のフルハンドシェイク（1-RTT）

### 4.1 メッセージフロー

```
Client                                    Server
──────                                    ──────
Key Share
ClientHello          ──────────→
                                          ServerHello
                                          {EncryptedExtensions}
                                          {CertificateRequest*}
                                          {Certificate}
                                          {CertificateVerify}
                     ←──────────          {Finished}
{Certificate*}
{CertificateVerify*}
{Finished}           ──────────→
[Application Data]   ←────────→          [Application Data]

  {} = ハンドシェイク鍵で暗号化
  [] = アプリケーション鍵で暗号化
  *  = オプション
```

TLS 1.3では、**1-RTT**でハンドシェイクが完了する。クライアントは最初のメッセージ（ClientHello）に鍵共有のパラメータを含めるため、サーバーは応答と同時に暗号化を開始できる。

### 4.2 詳細なフロー

**Step 1: ClientHello**

クライアントは以下の情報を送信する：
- サポートするTLSバージョン（`supported_versions`拡張）
- サポートする暗号スイート
- **Key Share**：ECDHE公開鍵（例：X25519の公開値）
- SNI（Server Name Indication）：接続先のホスト名

TLS 1.2では鍵交換パラメータはハンドシェイクの後半で交換されていたが、TLS 1.3ではClientHelloに含めることで1-RTTを実現する。クライアントはサーバーがどの鍵交換グループを選ぶかを予測して投機的にKey Shareを送信する。

**Step 2: ServerHello**

サーバーは以下を応答する：
- 選択した暗号スイート
- **Key Share**：サーバーのECDHE公開鍵

この時点で双方がECDHEの共有秘密を計算でき、以降のハンドシェイクメッセージは暗号化される。

**Step 3: 暗号化されたサーバーメッセージ**

- **EncryptedExtensions**：暗号化された拡張情報
- **Certificate**：サーバー証明書（X.509）
- **CertificateVerify**：サーバーの秘密鍵による署名（ハンドシェイクトランスクリプト全体に対する署名）
- **Finished**：ハンドシェイクの完全性を検証するMAC

**Step 4: クライアントの応答**

- **Finished**：クライアント側のハンドシェイク完了確認

### 4.3 鍵スケジュール

TLS 1.3の鍵導出は**HKDF（HMAC-based Key Derivation Function）**に基づく厳密なスケジュールで行われる。

```
             0
             │
             ↓
   PSK ──→ HKDF-Extract ──→ Early Secret
             │
             ↓
           Derive-Secret(., "derived", "")
             │
             ↓
(EC)DHE ──→ HKDF-Extract ──→ Handshake Secret
             │                    │
             │              ┌─────┴─────┐
             │              ↓           ↓
             │       client_hs_key  server_hs_key
             ↓
           Derive-Secret(., "derived", "")
             │
             ↓
   0 ────→ HKDF-Extract ──→ Master Secret
                                  │
                            ┌─────┴─────┐
                            ↓           ↓
                     client_app_key  server_app_key
```

鍵スケジュールの特徴：
- **段階的な鍵導出**：PSK（事前共有鍵）とECDHEの両方を入力とし、段階的に鍵を導出
- **トランスクリプトハッシュ**：ハンドシェイクメッセージ全体のハッシュが鍵導出に組み込まれ、改ざんを検出
- **鍵の分離**：ハンドシェイク用鍵とアプリケーション用鍵が分離される

## 5. 0-RTT（Early Data）

### 5.1 概要

TLS 1.3は、以前に接続したサーバーに再接続する場合、**0-RTT**でアプリケーションデータを送信できる。

```
Client                                    Server
──────                                    ──────
ClientHello
+ early_data
+ Key Share
(Application Data)   ──────────→
                                          ServerHello
                                          {EncryptedExtensions}
                                          {Finished}
                     ←──────────
{Finished}           ──────────→
[Application Data]   ←────────→          [Application Data]
```

クライアントは前回のセッションで取得した**PSK（Pre-Shared Key）**を使って、ClientHelloと同時にアプリケーションデータ（Early Data）を暗号化して送信する。

### 5.2 リプレイ攻撃のリスク

0-RTTの最大の問題は**リプレイ攻撃**に対して脆弱であることである。攻撃者がClientHelloとEarly Dataを記録し、サーバーに再送すると、サーバーはこれを正当なリクエストとして処理してしまう可能性がある。

これは、0-RTTデータが（ECDHEの共有秘密ではなく）PSKのみで暗号化されるため、前方秘匿性を持たないことに起因する。

対策：
- 0-RTTは冪等なリクエスト（GETなど）にのみ使用する
- サーバー側でリプレイ検出メカニズムを実装する（タイムスタンプ、使い捨てトークンなど）
- 金融取引や状態変更を伴うリクエストには0-RTTを使用しない

### 5.3 0-RTTの実用的な位置づけ

0-RTTはレイテンシに敏感なアプリケーション（CDN、検索エンジンなど）で有用だが、リプレイ攻撃のリスクを十分に理解した上で使用する必要がある。多くのサーバー実装では0-RTTをデフォルトで無効にしている。

## 6. TLS 1.3の暗号スイート

### 6.1 簡素化された暗号スイート

TLS 1.3では暗号スイートが**AEAD暗号 + ハッシュ関数**の2要素のみに簡素化された（鍵交換と署名は別途ネゴシエーション）：

| 暗号スイート | AEAD | ハッシュ |
|---|---|---|
| TLS_AES_128_GCM_SHA256 | AES-128-GCM | SHA-256 |
| TLS_AES_256_GCM_SHA384 | AES-256-GCM | SHA-384 |
| TLS_CHACHA20_POLY1305_SHA256 | ChaCha20-Poly1305 | SHA-256 |

TLS 1.2の数百の暗号スイートからわずか3つ（+2つのオプション）に削減された。

### 6.2 廃止された暗号

TLS 1.3で廃止されたもの：
- **RSA鍵交換**：前方秘匿性がない
- **静的DH/ECDH鍵交換**：前方秘匿性がない
- **CBC暗号モード**：パディングオラクル攻撃のリスク
- **RC4**：統計的偏り
- **SHA-1**：衝突が実証済み
- **圧縮**：CRIME攻撃
- **再ネゴシエーション**：複雑さと攻撃リスク
- **DSA署名**

## 7. ハンドシェイクの暗号化

TLS 1.2との大きな違いの一つが、ハンドシェイクメッセージの暗号化である。

| メッセージ | TLS 1.2 | TLS 1.3 |
|---|---|---|
| ClientHello | 平文 | 平文 |
| ServerHello | 平文 | 平文 |
| Certificate | 平文 | **暗号化** |
| CertificateVerify | 平文 | **暗号化** |
| Finished | 暗号化 | 暗号化 |

サーバー証明書が暗号化されることで、盗聴者はクライアントがどのサーバーに接続しているかを証明書から判断できなくなる（ただし、SNIは平文のまま。この問題はECH（Encrypted Client Hello）で対処される）。

## 8. TLS 1.2からの移行の実際

### 8.1 後方互換性

TLS 1.3はTLS 1.2との**後方互換性**を維持するように設計されている。ClientHelloのフォーマットはTLS 1.2と互換であり、`supported_versions`拡張でTLS 1.3をネゴシエーションする。

中間装置（ファイアウォール、ロードバランサーなど）がTLS 1.3のハンドシェイクを正しく処理できない場合があるため、TLS 1.3はServerHelloのフォーマットをTLS 1.2と似せるなどの互換性上の配慮がなされている。

### 8.2 ダウングレード攻撃の防止

TLS 1.3では、サーバーがTLS 1.2以下にダウングレードした場合にServerHelloのランダム値の末尾に特定のセンチネル値を埋め込む。TLS 1.3対応クライアントはこの値を検出し、ダウングレードが意図的か攻撃によるものかを判断できる。

### 8.3 現在の普及状況

2025年時点で、主要なWebブラウザ、Webサーバー、CDNはすべてTLS 1.3をサポートしている。Webトラフィックの大部分がTLS 1.3で保護されている。

## 9. ECH（Encrypted Client Hello）

### 9.1 SNIの問題

TLS 1.3でも**SNI（Server Name Indication）**は平文で送信される。SNIにはクライアントが接続しようとしているホスト名が含まれるため、盗聴者や検閲者はユーザーのアクセス先を把握できる。

### 9.2 ECHの仕組み

**ECH（Encrypted Client Hello）**は、ClientHelloの機密部分（SNIを含む）を暗号化する拡張である。DNSレコード（HTTPS RR）でサーバーのECH公開鍵を配布し、クライアントはこの鍵でClientHelloの内部を暗号化する。

ECHにより、盗聴者はクライアントがどのサーバーに接続しているかをTLSレベルで判断できなくなる。ただし、DNS over HTTPS（DoH）やDNS over TLS（DoT）と組み合わせなければ、DNSクエリからアクセス先が漏洩する。

## 10. TLS 1.3とポスト量子暗号

TLS 1.3の鍵交換にML-KEM（Kyber）を統合するハイブリッド方式が実装されつつある。X25519とML-KEM-768を組み合わせた**X25519Kyber768**が主要ブラウザで実験的にサポートされている。

ハイブリッド鍵交換では、ClientHelloのKey Shareサイズが従来の32バイトから約1200バイトに増大する。これはハンドシェイクのパケットサイズに影響するが、ほとんどの環境で許容範囲内である。

## 11. まとめ

TLS 1.3は、インターネットセキュリティの重要な進歩であり、以下の改善を実現した：

- **1-RTTハンドシェイク**：レイテンシの大幅な削減
- **前方秘匿性の強制**：すべての接続でECDHEを使用
- **レガシー暗号の排除**：安全でないアルゴリズムの完全廃止
- **ハンドシェイクの暗号化**：証明書を含むメタデータの保護
- **暗号スイートの簡素化**：AEAD暗号のみをサポート
- **0-RTT**：再接続時のさらなる高速化（リスクを理解した上で使用）

TLS 1.3は「正しいことを簡単にし、間違ったことを困難にする」という設計哲学を体現しており、安全なインターネット通信の新たな基盤となっている。
