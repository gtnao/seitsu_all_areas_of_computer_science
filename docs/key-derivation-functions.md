# 鍵導出関数（PBKDF2, bcrypt, Argon2）— パスワードハッシュの正しいやり方

## 1. はじめに：なぜSHA-256でパスワードをハッシュしてはいけないのか

Webアプリケーションにおいて、ユーザーのパスワードを安全に保存することは最も基本的なセキュリティ要件の一つである。パスワードを平文で保存するのは論外だが、単にSHA-256でハッシュするだけでも十分ではない。

その理由は、SHA-256が**高速に動作するように設計されている**からである。現代のGPUは毎秒数十億回のSHA-256計算が可能であり、辞書攻撃やブルートフォース攻撃によって、一般的なパスワード（8文字以下、辞書語、パターン化されたもの）は数秒から数分で解読されてしまう。

**鍵導出関数（KDF: Key Derivation Function）**、特にパスワードハッシュ関数は、この問題を解決するために設計された。意図的に計算を**遅く**し、攻撃者のコストを大幅に引き上げる。

## 2. パスワードハッシュの基本原則

### 2.1 ソルト（Salt）

**ソルト**は、各パスワードのハッシュ計算にユニークなランダム値を追加する仕組みである。

ソルトなしの場合：
- 同じパスワードは同じハッシュ値を持つ
- 事前計算テーブル（レインボーテーブル）による攻撃が有効
- 複数ユーザーが同じパスワードを使っていることが判明する

ソルトありの場合：
- 同じパスワードでも異なるソルトにより異なるハッシュ値になる
- レインボーテーブルが無効化される（ソルトごとにテーブルを作る必要がある）
- 各パスワードを個別に攻撃する必要がある

ソルトは**最低16バイト**のランダム値とし、各ユーザーのパスワードごとに異なる値を使用する。ソルトは秘密にする必要はなく、ハッシュ値と一緒に保存する。

### 2.2 ストレッチング（Key Stretching）

**ストレッチング**は、ハッシュ関数を多数回繰り返し適用することで、1回のハッシュ計算にかかる時間を意図的に長くする手法である。

例えば、PBKDF2で10万回のイテレーションを設定すると、1パスワードの検証に約100ミリ秒かかる。正規ユーザーにとっては許容範囲だが、攻撃者にとっては毎秒10回の試行しかできないことになる。

### 2.3 メモリハードネス

GPU/ASICは演算能力は高いが、メモリ帯域幅には制約がある。**メモリハード関数**は、大量のメモリを消費する計算を行うことで、GPU/ASICによる並列攻撃を困難にする。

これは比較的新しい概念であり、Argon2で本格的に導入された。

## 3. PBKDF2

### 3.1 概要

**PBKDF2（Password-Based Key Derivation Function 2）**はRFC 8018で定義されたパスワードベースの鍵導出関数であり、2000年にRSA Laboratoriesによって標準化された。

### 3.2 アルゴリズム

PBKDF2は、擬似ランダム関数（通常はHMAC）を繰り返し適用する：

$$
\text{DK} = T_1 \| T_2 \| \ldots \| T_l
$$

$$
l = \lceil dkLen / hLen \rceil
$$

各ブロック $T_i$ は以下のように計算される：

$$
T_i = U_1 \oplus U_2 \oplus \ldots \oplus U_c
$$

$$
U_1 = \text{HMAC}(\text{Password}, \text{Salt} \| \text{INT}(i))
$$

$$
U_j = \text{HMAC}(\text{Password}, U_{j-1}) \quad (j \geq 2)
$$

ここで $c$ はイテレーション回数、$\text{INT}(i)$ は $i$ の4バイトビッグエンディアン表現である。

```
Password, Salt
     │
     ↓
U₁ = HMAC(Password, Salt ‖ INT(1))
     │
     ↓
U₂ = HMAC(Password, U₁)
     │
     ↓
U₃ = HMAC(Password, U₂)
     │
     ↓
    ...（c回繰り返し）
     │
     ↓
T₁ = U₁ ⊕ U₂ ⊕ ... ⊕ Uc
```

### 3.3 パラメータ選択

| パラメータ | 推奨値 |
|---|---|
| PRF | HMAC-SHA-256 |
| イテレーション回数 | 600,000以上（2023年OWASP推奨） |
| ソルト長 | 16バイト以上 |
| 出力長 | 32バイト（SHA-256の場合） |

イテレーション回数は、正規ユーザーの認証時間が100〜500ミリ秒になるようにサーバーの性能に合わせて調整する。

### 3.4 利点と欠点

**利点**：
- 広く標準化されている（NIST、RFC）
- 実装がシンプル
- あらゆるプラットフォームで利用可能
- FIPS 140-2/3準拠

**欠点**：
- **メモリハードではない**：GPU/FPGA/ASICによる並列攻撃に対して脆弱
- 計算はほぼHMAC呼び出しのみであり、GPU上で効率的に並列実行できる
- 同じセキュリティレベルを達成するためにbcryptやArgon2よりも多くのイテレーションが必要

## 4. bcrypt

### 4.1 概要

**bcrypt**は1999年にNiels ProvosとDavid Mazièresによって設計された、Blowfish暗号をベースにしたパスワードハッシュ関数である。OpenBSDのデフォルトパスワードハッシュとして長年使われてきた。

### 4.2 設計思想

bcryptの重要な設計原則は**Eksblowfishアルゴリズム**にある。これはBlowfishの鍵スケジュールを拡張し、コストパラメータに応じて**反復的な鍵セットアップ**を行うことで、計算コストを調整可能にしている。

通常のBlowfishでは鍵セットアップは1回だけだが、Eksblowfishではこの処理を $2^{\text{cost}}$ 回繰り返す。鍵セットアップには4KBのS-Boxが必要であり、この**メモリ依存性**がGPU攻撃を困難にしている。

### 4.3 アルゴリズム

```
EksBlowfishSetup(cost, salt, password):
    state = InitialBlowfishState()
    state = ExpandKey(state, salt, password)
    for i = 1 to 2^cost:
        state = ExpandKey(state, 0, password)
        state = ExpandKey(state, 0, salt)
    return state

bcrypt(cost, salt, password):
    state = EksBlowfishSetup(cost, salt, password)
    ctext = "OrpheanBeholderScryDoubt"  // 192-bit constant
    for i = 1 to 64:
        ctext = EncryptECB(state, ctext)
    return Concatenate(cost, salt, ctext)
```

### 4.4 出力形式

bcryptのハッシュ値は以下の形式で保存される：

```
$2b$12$WApznUPhDubN0oeveSXHp.xUGJq6jKfF3tMHYvKvCOVFyJSqshxrC
 │  │  │                      │
 │  │  │                      └── ハッシュ値（Base64、31文字）
 │  │  └── ソルト（Base64、22文字、128ビット）
 │  └── コストファクター（2^12 = 4096回のイテレーション）
 └── アルゴリズムバージョン（2b）
```

### 4.5 パラメータ選択

| パラメータ | 推奨値 |
|---|---|
| コストファクター | 10〜14（$2^{10}$〜$2^{14}$） |
| ソルト長 | 128ビット（固定） |
| 出力長 | 184ビット（固定） |

コストファクターが1増えるごとに計算時間が2倍になる。一般的にはコスト10（約100ms）から始めて、サーバーの性能に合わせて調整する。

### 4.6 利点と欠点

**利点**：
- 20年以上の実績
- 4KBのS-Boxによる適度なメモリ依存性がGPU攻撃を抑制
- シンプルなコストパラメータ
- 広くサポートされている

**欠点**：
- **パスワード長が72バイトに制限**される（Blowfishの制約）
- メモリ使用量が固定（4KB）であり、現代の基準では少ない
- GPU攻撃の完全な防御には不十分（FPGAでの攻撃事例がある）
- 出力長が184ビット固定

## 5. scrypt

### 5.1 概要

**scrypt**は2009年にColin Percivalによって設計された、**メモリハード**な鍵導出関数である。暗号通貨（Litecoinなど）のProof-of-Workや、パスワードハッシュに使われている。

### 5.2 メモリハードネスの仕組み

scryptの核心は**Sequential Memory-Hard Function**の実現にある。大量のメモリに擬似ランダムなデータを書き込み、その後そのデータにランダムにアクセスする。メモリを使わずに計算しようとすると、計算量が大幅に増加する。

主要な内部関数は**ROMix**であり、以下のように動作する：

1. PBKDF2でパスワードから初期ブロックを導出する
2. 大量の中間ブロックを生成してメモリに格納する（Sequential Writes）
3. 中間ブロックにランダムアクセスして混合する（Random Reads）
4. 最終結果をPBKDF2で鍵に変換する

### 5.3 パラメータ

| パラメータ | 意味 | 推奨値 |
|---|---|---|
| $N$ | CPU/メモリコスト（2の冪） | $2^{15}$〜$2^{20}$ |
| $r$ | ブロックサイズ | 8 |
| $p$ | 並列度 | 1 |

メモリ使用量はおよそ $128 \times N \times r$ バイトとなる。$N = 2^{15}, r = 8$ の場合、約32MBのメモリを消費する。

### 5.4 欠点

- パラメータのチューニングが複雑（$N$, $r$, $p$ の3つ）
- メモリとCPUのバランスを個別に調整できない
- サイドチャネル攻撃への耐性が不十分な場合がある
- Argon2の登場により、新規システムではArgon2が推奨されるようになった

## 6. Argon2

### 6.1 概要

**Argon2**は2015年のPassword Hashing Competition（PHC）の優勝アルゴリズムであり、Alex Biryukovらによって設計された。現在、パスワードハッシュの**最推奨アルゴリズム**とされている（OWASP 2023推奨）。

### 6.2 バリエーション

| バリエーション | 特性 | 用途 |
|---|---|---|
| **Argon2d** | データ依存メモリアクセス | 暗号通貨、サイドチャネル攻撃のリスクがない環境 |
| **Argon2i** | データ独立メモリアクセス | パスワードハッシュ（サイドチャネル攻撃耐性） |
| **Argon2id** | 前半がArgon2i、後半がArgon2d | **推奨**。サイドチャネル耐性とGPU耐性のバランス |

### 6.3 アルゴリズムの概要

Argon2は大量のメモリを**ブロック**（1024バイト/ブロック）の配列として使用し、以下のフェーズで処理を行う：

1. **初期化**：パスワード、ソルト、パラメータからBlake2bハッシュを使って初期ブロックを生成
2. **メモリ充填**：ブロック配列を計算で埋める。各ブロックは前のブロックと、配列内のランダムな位置のブロックを参照して生成される
3. **追加パス**：パラメータに応じてメモリ充填を複数回繰り返す
4. **最終化**：最後の列のブロックをXORし、Blake2bで最終ハッシュ値を生成

```
┌─────┬─────┬─────┬─────┐
│ B₀₀ │ B₀₁ │ B₀₂ │ B₀₃ │  Lane 0
├─────┼─────┼─────┼─────┤
│ B₁₀ │ B₁₁ │ B₁₂ │ B₁₃ │  Lane 1
├─────┼─────┼─────┼─────┤
│ B₂₀ │ B₂₁ │ B₂₂ │ B₂₃ │  Lane 2
├─────┼─────┼─────┼─────┤
│ B₃₀ │ B₃₁ │ B₃₂ │ B₃₃ │  Lane 3
└─────┴─────┴─────┴─────┘
  Slice 0  Slice 1  Slice 2  Slice 3
```

メモリは**レーン**（並列処理の単位）と**スライス**（時間的な区分）に分かれる。各ブロックの生成時にランダムなブロックを参照するため、メモリの省略が困難になっている。

### 6.4 パラメータ

| パラメータ | 意味 | OWASP推奨（2023） |
|---|---|---|
| メモリ $m$ | 使用メモリ量（KiB） | 19456 KiB（19 MiB）以上 |
| イテレーション $t$ | パスの回数 | 2以上 |
| 並列度 $p$ | レーン数 | 1 |
| タグ長 | 出力のバイト数 | 32バイト |
| バリエーション | アルゴリズムの種類 | Argon2id |

推奨パラメータの調整方法（OWASPガイダンス）：

1. 利用可能なメモリ量を決定する
2. 並列度をCPUコア数の2倍に設定する
3. イテレーション回数を1から始めて、認証時間が許容範囲内に収まるまで増やす

### 6.5 Argon2がbcrypt/PBKDF2より優れる理由

1. **メモリハードネス**：大量のメモリを要求することで、GPU/ASICによる並列攻撃を根本的に困難にする
2. **柔軟なパラメータ**：メモリ、CPU、並列度を独立に調整できる
3. **トレードオフ耐性**：メモリを節約しようとすると計算量が急激に増大する
4. **サイドチャネル耐性**：Argon2id/Argon2iはデータ独立のメモリアクセスパターンを提供する
5. **現代的な設計**：PHCでの公開評価を経ており、多くの暗号学者によるレビューを受けている

## 7. 比較

| 特性 | PBKDF2 | bcrypt | scrypt | Argon2id |
|---|---|---|---|---|
| 設計年 | 2000 | 1999 | 2009 | 2015 |
| メモリハード | いいえ | 一部（4KB） | はい | はい |
| GPU耐性 | 低い | 中程度 | 高い | 高い |
| パラメータ柔軟性 | 低い | 低い | 中程度 | 高い |
| 標準化 | NIST, RFC | — | RFC 7914 | RFC 9106 |
| FIPS準拠 | はい | いいえ | いいえ | いいえ |
| 推奨度 | レガシー/FIPS | レガシー | 用途次第 | **最推奨** |

## 8. 実装のベストプラクティス

### 8.1 アルゴリズムの選択

- 新規システム：**Argon2id**
- FIPS準拠が必要：**PBKDF2-HMAC-SHA-256**
- レガシーシステム：bcryptでも許容（ただし移行を計画すべき）

### 8.2 パスワードのペッパー

**ペッパー（Pepper）**は、ソルトに加えてサーバー側で保持する秘密の値である。データベースが漏洩してもペッパーが漏洩しなければ、攻撃者はオフライン攻撃を行えない。

$$
\text{Hash} = \text{Argon2id}(\text{password} \| \text{pepper}, \text{salt}, \text{params})
$$

ペッパーはデータベースではなく、環境変数やHSM（Hardware Security Module）に保存する。

### 8.3 ハッシュのアップグレード

パスワードハッシュアルゴリズムは時間とともにアップグレードが必要になる。2つの戦略がある：

1. **ログイン時の再ハッシュ**：ユーザーがログインした際に、古いアルゴリズムのハッシュを検証し、新しいアルゴリズムで再ハッシュして保存する
2. **ラッピング**：既存のハッシュを新しいアルゴリズムでラップする（$\text{Argon2id}(\text{bcrypt\_hash}, \text{salt})$）。ログインを待たずに即座にアップグレードできる

### 8.4 DoS対策

パスワードハッシュは意図的に遅いため、攻撃者が大量のログインリクエストを送信してサーバーのCPU/メモリを枯渇させるDoS攻撃のリスクがある。

対策：
- レート制限（IP/アカウントベース）
- CAPTCHAの導入
- パスワードの最大長制限（bcryptの72バイト制限以外にも、極端に長いパスワードによるDoSを防ぐ）
- ハッシュ計算の非同期化（ワーカースレッドへのオフロード）

## 9. まとめ

パスワードハッシュは、ユーザー認証システムの安全性を支える最も重要な要素の一つである。SHA-256のような汎用ハッシュ関数を直接使うのではなく、専用のパスワードハッシュ関数を使うことが不可欠である。

推奨の優先順位：

1. **Argon2id**：最推奨。メモリハードネスとサイドチャネル耐性を兼ね備える
2. **bcrypt**：実績があり、広くサポートされている
3. **PBKDF2-HMAC-SHA-256**：FIPS準拠が必要な場合

いかなる場合も以下を守る：

- ソルトは最低16バイトのランダム値を使用する
- パラメータは認証時間が100〜500ミリ秒になるように調整する
- 計算コストは定期的に見直し、ハードウェアの進歩に合わせてアップグレードする
- タイミング攻撃を防ぐため、ハッシュ比較には定数時間比較を使用する

パスワードハッシュの技術は一見地味だが、データベースの漏洩が日常的に発生する現代において、ユーザーの最後の防衛線として極めて重要な役割を果たしている。
